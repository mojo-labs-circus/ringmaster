## Architecture Rules
- Repository pattern everywhere вЂ” nodes never touch raw SQL or storage directly
- Never call `get_auth_repository()`, `get_task_repository()`, or `get_history_repository()` inside a function body вЂ” repository instantiation always happens at module level or via FastAPI's `Depends()` injection
- All data operations scoped to `user_id` вЂ” no global queries
- Model names always from `config.yaml` via `config.py` вЂ” never hardcoded
- Secrets via env vars only вЂ” `JARVIS_SECRET_KEY` hard-fails if unset
- One branch per phase вЂ” merge to main only when complete and verified
- Write tests alongside implementation вЂ” never defer
- Comment the why, not the what

## WebSocket Frame Contract
All frames are typed JSON with a `message_id` field:

    {"type": "token",  "message_id": "abc123", "content": "..."}
    {"type": "done",   "message_id": "abc123", "refresh": []}
    {"type": "error",  "message_id": "abc123", "message": "..."}
    {"type": "status", "message_id": "abc123", "message": "..."}

Client sends:

    {"message_id": "abc123", "content": "user message here"}

message_id is generated by the client using `secrets.token_hex(4)` вЂ” short random string,
not a UUID.

## Key Config
- `config.yaml` вЂ” non-sensitive config, single source of truth
- `config.py` вЂ” only file that reads config.yaml and env vars
- `.env` вЂ” secret values, never committed
- `JARVIS_SECRET_KEY` вЂ” required always, hard-fails if unset
- `JARVIS_DB_BACKEND` вЂ” `sqlite` in dev, `postgres` in production
- `JARVIS_DB_URL` вЂ” only needed when backend is postgres
